Introduction to the ER Model

The Entity Relationship (ER) Model is a conceptual data modeling technique used to visually represent the structure of a database. It shows the major data objects (entities), their characteristics (attributes), and how they relate to one another (relationships).

The ER Model was introduced by Peter Chen in 1976 and is widely used in database design before implementing a database in SQL. The diagrams based on the model are called ER Diagrams (ERDs).


Components of the ER Model

1. Entity - An entity is any real-world object, person, place, event, or concept about which data is to be stored. Examples: Student, Course, Employee, Vehicle, Library Book.

Types of Entities

i. Strong Entity - An entity that has a primary key of its own and can stand independently. E.g Student, Department.

ii. Weak Entity - An entity that cannot be uniquely identified without another entity (strong entity). E.g Dependent (child) of an Employee.


2. Attributes - Attributes are the properties or characteristics of an entity. Examples: Student, Course, Employee, Vehicle, Library Book.

Types of Attributes

i. Simple Attribute - They cannot be divided further. E.g Age, Gender.

ii. Composite Attribute - They can be broken into sub-parts. E.g Example: Full Name → First Name + Middle Name + Last Name.

iii. Derived Attribute - They are calculated from another attribute. Example: Age derived from Date of Birth.

iv. Multi-valued Attribute - They are attributes with more than one value. Example: Phone Numbers, Email Addresses.

iv. Key Attribute - Attributes that uniquely identifies an entity instance. Example: Student_ID, Employee_ID.

3. Relationships - A relationship describes how entities interact or are connected.

Types of Relationships (Based on Cardinality)

i. One-to-One (1:1) - One entity relates to at most one entity. Example: One student has one school.

ii. One-to-Many (1:N) - One entity relates to many entities. Example: One student has many subjects.

iii. Many-to-Many (M:N) - Many entities relate to many entities. Example: Students register for many subjects; subjects have many students.

Steps in Creating an ER Diagram

i. Identify all entities (e.g., Student, Course).

ii. Identify attributes of each entity.

iii. Determine primary keys.

iv. Establish relationships between entities.

v. Determine cardinality (1:1, 1:N, M:N).

vi. Draw the ER Diagram with appropriate symbols.

vii. Review and refine for completeness.


Conversion of ER Model to Relational Model - ER diagrams are usually transformed into database tables (Relational Model) following a set of rules:

i. Each entity → becomes a table.

ii. Each attribute → becomes a column.

iii. Primary key is kept.

iv. Relationships

→ 1:N: Place foreign key on the N-side table.

→ 1:1: Foreign key can go on either side.

→ M:N: Create a junction table (associative table).

v. Weak entities become tables with foreign key + discriminator.


Advantages of the ER Model

i. Easy to understand and visualize.

ii. Simplifies database design.

iii. Helps communicate ideas between developers and stakeholders.

iv. Reduces data redundancy.

v. Forms a strong foundation for relational databases.
